## ⚠️ Why this file is split in two

Every fact in this file belongs in exactly one of two halves, and the split is the entire point of this document:

- **Derived** facts come from somewhere else — a ticket tracker, a set of accounts, a codebase, a calendar. If a derived fact turns out to be wrong, that's a **sync problem**: the source moved on and this file didn't follow. The fix is mechanical — go re-check the source and update the transcription. No judgment call required.
- **Judgment** facts exist only in the project owner's head — current priorities, which option they're leaning toward, whether a relationship is still active, what's being deferred and why. If a judgment fact turns out to be wrong or outdated, that's not a sync problem — it's a **signal** that a decision needs to be revisited. There is no "source" to re-check; the owner has to actually decide again.

Mixing these together is exactly how staleness goes silent: an out-of-date derived fact and an out-of-date decision read identically if they're written in the same paragraph, and the reader can't tell whether the fix is "go look something up" or "go make a call." Keeping them apart means every stale entry tells you, at a glance, what kind of action it needs.

When adding a new line to this file, ask: *if this turns out wrong next month, will it be because a source changed, or because I changed my mind?* That answer decides which half it goes in.
